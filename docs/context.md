# Status / Context

## Цель / видение
PcBench — облачный сервис тестирования, анализа производительности и выявления узких мест ПК с локальным агентом и веб-интерфейсом. Цель — дать пользователю быстрый и понятный ответ, насколько его сборка подходит под разные сценарии (обучение, игры, программирование, офис) и что улучшить в первую очередь. Сервис должен работать без сложной настройки, быть воспроизводимым по результатам, и давать рекомендации на базе реальных метрик и спецификаций компонентов.  
В перспективе — стать точкой принятия решений о покупке/апгрейде и каналом монетизации через подписку и аффилиат-переходы.

## Текущий фокус
* Доведение MVP анализа сборки для анонимных пользователей (подбор комплектующих → анализ → рекомендации по 4 сценариям).
* Уточнение API/контрактов между фронтом и бэком под анонимный сборщик.
* Стабилизация CI/CD и окружений (dev/stage/beta/prod) для регулярных поставок.
* Инфраструктурная схема окружений и пайплайнов деплоя (Timeweb Cloud + Kubernetes).
* Фиксация гипотезы упаковки и цен (Free/Pro/Creator/B2B) как базы для экспериментов 2026: `monetization/pricing_packaging.md`.

## Что готово
* Базовая документация по модулям (Agent/Backend/Frontend/ML/Infrastructure/SLA).
* Концепт рекомендации/скоринга сборки (BalancedScoreEngine) в бэкенде (в разработке/итерации).

## Блокеры / риски
* Неопределённость публичного API для подбора комплектующих (нужно безопасно открыть для анонимных пользователей без злоупотреблений).
* Качество и полнота спецификаций компонентов в БД (влияет на точность рекомендаций).
* Разрастание различий окружений (dev/stage/beta/prod) без формализации IaC/шаблонов.
* ML-часть может давать нестабильные результаты при нехватке данных/дрейфе.

## Ближайшие 1–2 недели (оперативный горизонт)
1) Зафиксировать контракты API для анонимного сборщика (список компонент, фильтры, пагинация, анализ сборки).
2) Доделать AnalyzeResponseDto и формат рекомендаций по 4 сценариям (структура, веса, текстовые подсказки).
3) Минимальные ограничения/защита публичных эндпойнтов (rate limit, кеширование справочников, audit).
4) Обновить документацию: ADR по окружениям и деплою; ADR по скорингу рекомендаций; ADR по монетизации.
